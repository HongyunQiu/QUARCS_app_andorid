## 2. 迁移后的整体结构

迁移完成后，Android 端不再依赖外部网页服务器来加载主前端，而是改为：

1. `QUARCS_app` 内部启动本地 HTTP 服务。
2. `WebView` 打开 `127.0.0.1` 上的内嵌页面入口。
3. 前端通过本地 WebSocket 连接 Android 内部的兼容命令服务。
4. 兼容命令服务继续保持原有 `Vue_Command` / `QT_Return` 协议风格。
5. 后续 Android 原生硬件实现只替换协议背后的具体服务，不需要先推翻前端协议。

当前运行时大致如下：

- 本地静态资源服务：`http://127.0.0.1:18080/`
- 本地兼容命令服务：`ws://127.0.0.1:8600`
- 页面入口：`http://127.0.0.1:18080/index.html`

## 3. Android 侧新增或关键改动

### 3.1 本地静态资源服务

关键文件：

- `QUARCS_app/embeddedassethttpserver.h`
- `QUARCS_app/embeddedassethttpserver.cpp`

作用：

- 从 App 内资源目录读取前端构建产物
- 通过本地 HTTP 对 `index.html`、JS、CSS、WASM、图片等资源提供访问
- 动态返回 `quarcs-runtime-config.js`
- 现在也负责从打包后的单文件瓦片包中读取离线地图瓦片

### 3.2 宿主运行时配置注入

关键文件：

- `QUARCS_app/apphostservice.h`
- `QUARCS_app/apphostservice.cpp`

作用：

- 生成运行时脚本 `quarcs-runtime-config.js`
- 向前端注入内嵌模式参数
- 提供时间、位置、语言、Wi-Fi 名称、版本号等初始状态

### 3.3 本地兼容命令服务

关键文件：

- `QUARCS_app/compatibilitycommandserver.h`
- `QUARCS_app/compatibilitycommandserver.cpp`

作用：

- 在 Android 内部启动本地 WebSocket 服务
- 接收前端发送的 `Vue_Command`
- 向前端返回 `QT_Return`
- 在过渡阶段尽量保持和原 `QUARCS_QT-SeverProgram` 的命令契约兼容

### 3.4 按硬件域拆分的兼容服务

当前已经预留了按域拆分的服务骨架，便于后续逐步把旧服务端能力迁入 Android：

- `QUARCS_app/devicegatewayservice.*`
- `QUARCS_app/imagingservice.*`
- `QUARCS_app/mountservice.*`
- `QUARCS_app/guiderservice.*`
- `QUARCS_app/focuserservice.*`
- `QUARCS_app/filterwheelservice.*`
- `QUARCS_app/platesolveservice.*`
- `QUARCS_app/scheduleservice.*`

这些服务当前以兼容和打通链路为主，后续可以逐个替换为 Android 原生实现。

### 3.5 QML 与入口控制

关键文件：

- `QUARCS_app/main.cpp`
- `QUARCS_app/main.qml`

作用：

- 启动内嵌 HTTP 服务与兼容命令服务
- 把宿主能力暴露给 QML
- 控制是否默认进入内嵌前端
- 管理 `WebView` 的打开、关闭与运行时状态同步

## 5. 构建与打包链路

当前 Android 构建已经和前端构建串联：

1. 运行 `QUARCS_app/build_android.sh`
2. 调用 `QUARCS_app/prepare_embedded_webui.sh`
3. 转到 `QUARCS_stellarium-web-engine/apps/web-frontend`
4. 执行 `npm run build:android-embedded`
5. 前端生成 `dist/`
6. 构建产物同步到：
   - `QUARCS_app/android-sources/assets/webui/`
   - `QUARCS_app/build-qt6-android-arm64/android-build/assets/webui/`
7. Qt 原生库重新编译
8. Gradle 打包 APK

关键脚本：

- `QUARCS_app/build_android.sh`
- `QUARCS_app/prepare_embedded_webui.sh`
- `QUARCS_stellarium-web-engine/apps/web-frontend/scripts/build-android-embedded.sh`

## 6. 离线地图处理方式

### 6.1 原始数据位置

离线地图瓦片原始目录：

- `QUARCS_stellarium-web-engine/apps/web-frontend/tile-server/tiles`

### 6.2 为什么不能直接把散文件打进 APK

原始瓦片文件数量非常多，直接按散文件打进 APK 时，会触发 APK ZIP 条目数上限，导致打包失败。

### 6.3 当前采用的过渡方案

当前改为在前端打包阶段，把瓦片整理成两个文件：

- `offline-tiles.pack`
- `offline-tiles-index.json`

然后与其他前端静态资源一起进入：

- `QUARCS_app/android-sources/assets/webui/`
- `QUARCS_app/build-qt6-android-arm64/android-build/assets/webui/`

运行时由 `EmbeddedAssetHttpServer` 根据请求路径查索引，再从 `offline-tiles.pack` 中按偏移读取对应 PNG 数据返回。

这使得：

- Android 仍能通过 `/tiles/...` 路径访问离线地图
- APK 不会因为瓦片散文件太多而打包失败

## 7. 过渡期兼容策略

本次迁移不是一次性把旧服务端全部删除，而是采用“宿主切换 + 协议兼容 + 服务渐进迁移”的方式。

核心思路：

- 前端先迁入 Android 包内运行
- 命令协议尽量保持不变
- 旧服务端中的硬件逻辑，未来按模块逐步迁移到 Android 原生服务
- 前端尽可能少感知底层实现是旧服务端还是 Android 原生实现

这样做的好处：

- 迁移风险更低
- 能边迁移边验证
- 前端和设备协议层不会频繁反复改动

## 8. 当前产物与验证结论

当前已完成的结果：

- Android 可直接加载内嵌前端，不再依赖外部网页服务
- 本地 HTTP 资源服务与本地 WebSocket 兼容服务已打通
- 前端主要资源、`opencv.js`、WASM、`skydata` 已纳入内嵌打包链路
- 离线地图已通过单文件瓦片包方案纳入 APK
- 用户侧已完成真机测试验证

当前 APK 调试包路径：

- `QUARCS_app/build-qt6-android-arm64/android-build/build/outputs/apk/debug/android-build-debug.apk`

## 9. 后续维护建议

后续如需继续推进这条路线，建议遵循下面的分工：

- 改前端界面和业务逻辑：优先在 `QUARCS_stellarium-web-engine/apps/web-frontend` 修改
- 改 Android 宿主、HTTP 承载、QML 入口：在 `QUARCS_app` 修改
- 改协议兼容与硬件迁移：优先在 `CompatibilityCommandServer` 及各 `*Service` 中推进
- 改地图或离线资源打包：优先看 `build-android-embedded.sh` 与 `EmbeddedAssetHttpServer`
