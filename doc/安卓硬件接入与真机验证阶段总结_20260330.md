## 3. Android USB 串口接入过程

### 3.1 最初问题

最开始 Android 端尝试直接打开 `/dev/ttyACM0`，日志表现为：

- `errno 13`
- `Failed to open focuser transport`

这说明 Android app 进程没有权限直接裸开设备节点。

### 3.2 解决方案

改为通过 Android `UsbManager` 申请 USB Host 权限，并通过 Java bridge 负责设备打开、配置、读写。

关键文件：

- [UsbSerialBridge.java](/home/q/workspace/QUARCS_app_andorid/android-sources/src/org/qtproject/example/QUARCS_app/UsbSerialBridge.java)
- [usb_serial_transport.cpp](/home/q/workspace/QUARCS_app_andorid/hardware/transports/usb_serial_transport.cpp)

### 3.3 CDC ACM 与 CH34x

初版 bridge 仅适配标准 CDC ACM。

在切换到 OnStep 后，日志显示设备为：

- `vid=0x1a86 pid=0x7523`

这是常见的 `CH340/CH341` USB 转串口。

当时错误为：

- `SET_LINE_CODING failed`

随后在 Java bridge 中补充了 `CH34x` 初始化路径。补完后：

- 串口打开成功
- `UsbSerialTransport opened via Android USB bridge "/dev/ttyACM0" baud 9600`

这一步是 OnStep 真机接入能够继续推进的关键拐点。

## 8. OnStep 真机验证结果

### 8.1 原始探测

为确认握手失败是否真的是协议问题，新增了 `MountProbeRaw`。

验证时直接探测底层命令，得到真实回包：

- `:GVP# -> On-Step`
- `:GR# -> 00:47:28`
- `:GD# -> -90*00:00`
- `:GU# -> Npz/ET260`

这证明：

- 串口链路通
- OnStep 设备确实在回复
- 早先的失败主要是本地握手判定过严

### 8.2 握手修正

修正后允许 `On-Step` 被识别为有效握手。

同时参照 INDI 对 `:GU#` 的判定，修正了 `Idle/Tracking/Slewing` 的粗糙解析。

### 8.3 已验证能力

在真机上已完成以下验证：

- `MountConnect` 成功
- `MountSelfTest` 成功
- `RA/DEC` 读取成功
- `Slew Rate` 切换成功
- `Goto` 基础链路成功
- `MoveWest` 成功
- `MoveEast` 成功
- `Abort` 成功

### 8.4 速度 9 往返运动测试

在用户旁观下，已执行真实运动测试：

1. 将速度切到 `9`
2. 向一个方向运动 `10s`
3. `Abort`
4. 向反方向运动 `10s`
5. `Abort`

回读数据表明：

- 运动过程中 RA/DEC 显著变化
- 两段运动都真实发生
- 停止命令有效

测试末尾 `:GU#` 返回：

- `nNpz/ET292`

### 8.5 当前残留问题

虽然真实运动已经成功，但 `Tracking` 的显示状态仍然存在一定不稳定：

- 业务日志可能提示 `Tracking on`
- 紧接着 `getMountParameters` 里的 `TelescopeTrack` 仍可能显示 `OFF`

这更像是当前统一状态模型对 OnStep `:GU#` 的映射仍然不够细，不影响这轮真实运动，但后续值得继续修。
