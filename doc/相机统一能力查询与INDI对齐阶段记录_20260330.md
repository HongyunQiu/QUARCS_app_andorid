# 相机统一能力查询与 INDI 对齐阶段记录

本文档用于记录 `QUARCS_app_andorid` 在 2026-03-30 针对相机统一接口继续推进的一轮工作。本轮重点不是新增厂商特性，而是将相机控制继续收敛到更接近 `INDI CCD` 常见能力模型的抽象上，为后续接入其他相机预留稳定接口边界。

## 1. 本轮目标

本轮工作的目标是：

- 在统一相机接口中增加“能力查询”层
- 优先采用接近 INDI 常见能力组织方式的抽象
- 避免把 `QHYCCD` 私有控制直接抬到统一接口
- 保持当前已经打通的真机拍摄链路不回退

当前遵循的原则是：

- 统一层优先保留通用能力
- 厂商差异优先收敛在 `core/adapter/sdk bridge`
- 如果后续出现明显超出 INDI 常见模型的新需求，再单独讨论是否扩统一接口

## 2. 本轮新增的统一能力模型

### 2.1 新增统一数据结构

在相机统一接口中新增了两类通用结构：

- `CameraControlRange`
  - 表达某个控制项是否支持
  - 表达最小值、最大值、步进、当前值、单位
- `CameraCapabilities`
  - 表达相机支持哪些能力
  - 表达常用控制项的范围
  - 表达读模式列表、帧类型列表、最大 binning

相关文件：

- [camera_device_interface.h](/home/q/workspace/QUARCS_app_andorid/hardware/devices/camera_device_interface.h)

当前统一能力模型包含：

- `exposureSupported`
- `gainSupported`
- `offsetSupported`
- `coolerSupported`
- `targetTemperatureSupported`
- `roiSupported`
- `binningSupported`
- `frameTypeSupported`
- `readModesSupported`
- `maxBinX`
- `maxBinY`
- `exposureSeconds`
- `gain`
- `offset`
- `targetTemperature`
- `readModeNames`
- `frameTypes`

### 2.2 统一接口新增查询入口

`ICameraDevice` 新增：

- `capabilities() const`

这意味着以后不管接的是 `QHYCCD`、别的 Android 厂商 SDK，还是未来从别的驱动迁移过来的相机实现，都需要提供一套统一的能力说明。

### 3.2 当前 QHYCCD 能力查询来源

`QhyccdCameraSdkBridge` 当前会使用下列 SDK 能力来填统一结构：

- `GetQHYCCDParamMinMaxStep`
  - 查询曝光、增益、偏置、目标温度等范围
- `IsQHYCCDControlAvailable`
  - 查询某项功能是否支持
- `GetQHYCCDNumberOfReadModes`
  - 查询读模式数量
- `GetQHYCCDReadModeName`
  - 查询读模式名称

当前已经映射到统一层的项目有：

- 曝光时间范围
- 增益范围
- 偏置范围
- 目标温度范围
- 是否支持温控
- 是否支持 ROI
- 是否支持 binning
- 是否支持帧类型
- 是否支持读模式
- 读模式名称列表

### 3.3 暂时没有进入统一接口的内容

以下内容当前没有进入统一层设置接口，而是保留在 `QHYCCD` 适配内部：

- `USB traffic`
- `DDR`
- Android USB prime / firmware / re-enumeration 处理
- `debayer` 和 Android 单帧下的 raw-only 限制细节

这样做是为了避免统一层过早被 `QHYCCD` 细节污染。

## 4. 服务层与调试页新增能力

- `CameraControlRange:Offset`

这套返回风格虽然不是直接复刻 INDI property，但组织思路是接近 INDI 常用 CCD 属性族的：

- 一组能力标记
- 一组控制范围
- 一组枚举列表

### 4.2 曝光语义继续向 INDI 习惯靠拢

本轮同时保留了上一轮对曝光语义的整理：

- `takeExposure` 默认跟随当前 `FrameType`
- `Light/Flat` 视为 light frame
- `Dark/Bias` 视为 dark frame
- 日志中明确记录秒值和微秒值

这样可以减少业务层显式携带“light/dark”布尔值的需要。

## 6. 当前接口边界判断

截至本轮结束，统一相机接口中我认为适合保留的通用能力包括：

- 连接 / 断开
- 状态刷新
- 曝光 / 中止 / 读帧
- 增益
- 偏置
- ROI
- binning
- 帧类型
- 冷却开关
- 目标温度
- 能力查询
- 参数范围查询
- 读模式列表查询

当前我认为不适合直接进入统一接口、应继续保留在厂商适配内部的内容包括：

- `USB traffic`
- `DDR`
- 固件下载与重枚举细节
- Android USB 权限与 fd prime 逻辑
- 厂商特定的图像总线优化项

如果未来业务侧确实需要控制这些项目，再单独讨论是否需要引入“高级厂商特性接口”。

## 7. 下一步建议

在当前边界下，下一步最合适继续做的是：

1. 用真机确认 `getCameraCapabilities` 的返回是否和 `QHY247C` 实际能力一致。
2. 基于这套统一能力模型，整理前端或业务层对“可用控件”的启用逻辑。
3. 再往后可以考虑补：
   - 当前读模式的回读
   - 当前读模式切换
   - 更完整的温控状态，例如制冷功率

其中第 3 点已经开始接近“超过当前统一层最小必需能力”的边界，尤其是制冷功率这类项目，是否放入统一接口建议在继续实现前先讨论。

## 8. 结论

本轮工作的结果是：

- 相机统一架构继续向 `INDI CCD` 常见能力模型靠拢
- 新增了统一能力与范围查询层
- `QHYCCD` 的能力查询已经被约束在通用模型内，而不是直接泄漏厂商特性
- 当前拍摄链路、显示链路与构建链路保持可用

这使得后续接入其他相机时，可以优先围绕统一能力模型来实现，而不必从 `QHYCCD` 专有调用方式重新倒推一套接口。
